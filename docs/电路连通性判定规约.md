# 电路连通性判定规约

> 配套文档：《电工模拟器网页产品需求文档 PRD》第 11 节。
> 本规约是模拟引擎的实现级规格，目标是让工程师不靠猜就能写出引擎。
> 第一版只做“状态模拟”：判断通 / 断、得电 / 失电、亮 / 灭、转 / 停、错 / 对，不算电流、电压、功率、波形。

---

## 0. 设计原则

1. **无方向**：端子的 `in/out` 只是命名，电气上双向导通；引擎不依赖端子方向。
2. **基于电气节点，不基于导线**：先把导线相连的端子合并成节点，再在节点图上算连通。
3. **导通 = 图的边**：每个“当前导通”的元件是一条边；状态变化就是边的增删。
4. **负载是短路判定的核心**：状态模拟无电阻，靠“路径是否经过负载”区分短路与正常回路。
5. **迭代到稳定**：含反馈的电路（自锁/互锁）多轮计算，必须有最大轮次与振荡检测。
6. **确定性**：同一电路 + 同一组用户操作，结果必须可复现，不依赖遍历顺序。

## 1. 输入数据模型

引擎每一轮接收一份电路快照（与 PRD §13 数据结构对应）：

```ts
type Vec2 = { x: number; y: number };

interface Terminal {
  id: string;            // 端子在元件内唯一，如 "A1" / "in" / "L1"
  offset: Vec2;          // 相对元件原点的坐标（UI 用，引擎不关心）
}

interface Component {
  id: string;            // 全局唯一，如 "km1"
  type: ComponentType;   // 见 §3
  name: string;          // 如 "KM1"
  position: Vec2;
  terminals: Terminal[];
  state: ComponentState; // 可变：pressed / closed / energized / faulted ...
  rules: {
    isLoad: boolean;     // 是否负载（灯泡/指示灯/电机/线圈 = true）
    [k: string]: unknown;
  };
  groupId?: string;      // 接触器/热继：线圈与其触点同属一组（见 §4）
}

interface Wire {
  id: string;
  from: { componentId: string; terminal: string };
  to:   { componentId: string; terminal: string };
}

interface Circuit {
  schemaVersion: number;
  components: Component[];
  wires: Wire[];
}
```

**端子的全局标识**用 `componentId#terminalId`（下称 *pin*），例如 `km1#A1`。

## 2. 引擎单轮流程总览

```
buildNodes(circuit)            // 2.1 并查集合并 pin → 电气节点
  → buildEdges(circuit, state) // 2.2 当前导通的元件 = 边
  → energize()                 // 2.3 标记带电节点 / 判定负载工作
  → detectShort()              // 2.4 短路判定
  → deriveNextState()          // 2.5 由“线圈得电”推导触点新状态
```

整体由 §6 的收敛循环反复调用，直到状态稳定。

### 2.1 生成电气节点（Union-Find）

1. 每个 pin 初始为独立集合。
2. 对每条 wire，`union(from_pin, to_pin)`。
3. 合并后，`find(pin)` 的代表元即该 pin 所属的**电气节点 id**。
4. 一个 pin 接多根线 → 这些线另一端的 pin 全部并入同一节点（天然支持并联）。

> 复杂度 O((P + W)·α)。P=pin 数，W=导线数。

### 2.2 生成边（当前导通的元件）

遍历所有“双端可导通元件”，按 §3 的导通条件判断当前是否导通：

- 导通 → 在它两个端子所属节点之间加一条无向边，记录 `isLoad`。
- 不导通 → 不加边。

得到无向图 `G = (Nodes, Edges)`，每条边带 `isLoad` 标志。

### 2.3 带电与负载工作判定

见 §5。

### 2.4 短路判定

见 §5.3。

### 2.5 状态推导

见 §4（接触器）、§3（各元件由自身/用户操作决定的状态）。

## 3. 元件导通规则表

| type | 端子 | isLoad | 当前导通条件 | 状态来源 |
|---|---|---|---|---|
| `wire`（隐式） | — | — | 恒通（已在 2.1 合并） | — |
| `single_phase_power` | L, N | — | 不是“边”，是电源极（见 §5.1） | 用户：有电/断电 |
| `three_phase_power` | L1,L2,L3,N,PE | — | 同上（三相极，见 §5.4） | 用户：有电/断电 |
| `breaker`（空开） | in_L,in_N,out_L,out_N | false | 合闸→导通；分闸→断开 | 用户点击 |
| `fuse`（熔断器） | in, out | false | 正常→导通；熔断→断开 | 故障态，见 §7 |
| `thermal_main`（热继主回路） | in, out | false | 正常→导通；动作→断开 | 故障态，见 §7 |
| `thermal_nc`（热继控制常闭） | in, out | false | 正常→导通；动作→断开 | 跟随热继故障态 |
| `switch`（普通开关） | in, out | false | 合→导通；断→断开 | 用户点击切换 |
| `button_no`（按钮常开） | in, out | false | 按下→导通；松开→断开 | 用户按住 |
| `button_nc`（按钮常闭） | in, out | false | 按下→断开；松开→导通 | 用户按住 |
| `contactor_coil`（线圈） | A1, A2 | **true** | 恒为“可通”的负载边（是否得电由 §5.2 判定） | 派生 |
| `contactor_main`（主触点） | L1/T1, L2/T2, L3/T3（三对） | false | 同组线圈得电→导通 | 派生（§4） |
| `contactor_no`（辅助常开） | in, out | false | 同组线圈得电→导通 | 派生（§4） |
| `contactor_nc`（辅助常闭） | in, out | false | 同组线圈失电→导通 | 派生（§4） |
| `lamp`（灯泡） | L, N | **true** | 恒为负载边（亮否由 §5.2 判定） | 派生 |
| `indicator`（指示灯） | L, N | **true** | 同灯泡，但不参与练习通过判定 | 派生 |
| `motor`（电机） | U, V, W | **true** | 三相负载（运行否由 §5.4 判定） | 派生 |

> 主触点有三对端子，应建模为**三条独立的边**（L1↔T1、L2↔T2、L3↔T3），同组线圈得电时三条边同时导通。

## 4. 接触器 / 热继的“组”关系

接触器是“一个线圈 + 多个触点”的联动体。用 `groupId` 把它们绑定：

- 线圈 `contactor_coil` 与其 `contactor_main` / `contactor_no` / `contactor_nc` 共享同一 `groupId`。
- 派生规则（每轮 2.5 执行）：
  - 线圈得电 → 该组所有主触点、辅助常开触点导通；辅助常闭触点断开。
  - 线圈失电 → 主触点、辅助常开断开；辅助常闭导通。
- 热继电器：`thermal_main`（主回路）与 `thermal_nc`（控制回路常闭）共享 `groupId`；过载动作时两者一起进入断开/动作态（动作判定见 §7）。

> 第一版线圈与触点是否同组由练习模板 / 用户在属性面板里指定（如“此辅助触点属于 KM1”）。自由模式下默认按元件命名前缀（KM1）归组。

## 5. 带电与工作判定

### 5.1 电源极

- 单相电源（有电）：其 `L` pin 所属节点标为 **L 极**，`N` pin 所属节点标为 **N 极**。
- 电源断电：不提供任何极，全图视为无电。
- 多个电源：各自的 L/N 极取并集（第一版练习一般单电源）。

### 5.2 单相负载是否工作

对每个单相负载（lamp / indicator / coil），设其两端子节点为 `a`、`b`：

> 工作 ⟺ `a` 与某个 L 极节点连通，`b` 与某个 N 极节点连通（或对称：a↔N 且 b↔L）。
> “连通”= 在图 G 上仅经过当前导通的边可达；**且该路径上不要求经过其它负载**（负载本身就是终点）。

实现：从所有 L 极节点做一次 BFS/DFS 得到 `reachL` 集合；从所有 N 极节点做一次得到 `reachN` 集合。负载工作 ⟺ `(a∈reachL ∧ b∈reachN) ∨ (a∈reachN ∧ b∈reachL)`。

> 注意：计算 `reachL/reachN` 时，负载边**也算可通行的边**（电流确实流过负载）。短路判定才区分负载（§5.3）。

### 5.3 短路判定

构造**子图 G′**：只保留 `isLoad=false` 的边（即去掉所有负载边）。

> 短路 ⟺ 在 G′ 上，某个 L 极节点与某个 N 极节点连通（L 不经过任何负载就直通 N）。
> 三相：任意两相极节点在 G′ 上连通，亦判短路。

短路时：标记为危险错误，阻止相关回路被判为“正常运行”，并触发 §7 的故障流程（提示 + 要求改线/复位）。

### 5.4 三相电机是否运行

主回路无 N，单独判定。设电机端子节点 U、V、W，三相电源极节点 L1、L2、L3：

> 运行 ⟺ U、V、W 分别与 L1、L2、L3 连通（一一对应，不强制相序，只要三相都通）。
> 实现：对三相各做一次可达集，要求 U∈reach(L1)、V∈reach(L2)、W∈reach(L3)（或任意相序的一个匹配存在）。

**简化电机**（练习用单相电源驱动时）：退化为单相负载，按 §5.2 判定，由练习模板声明 `motorMode: "simplified"`。

## 6. 收敛循环（含反馈电路）

```
function simulateToStable(circuit):
    MAX_ITER = 20
    history = []                       # 记录每轮的“状态指纹”
    for i in 1..MAX_ITER:
        nodes = buildNodes(circuit)
        G     = buildEdges(circuit)
        reachL, reachN = energize(nodes, G)
        coilStates = evalCoils(reachL, reachN)   # 5.2：哪些线圈得电
        shorts = detectShort(nodes, G)           # 5.3
        next   = deriveNextState(coilStates, faults)  # §4 派生触点 + §7 故障
        fp = fingerprint(next)         # 所有元件可变状态的哈希
        if fp == lastFingerprint:      # 与上一轮相同 → 已稳定
            return { stable: true, ... }
        if fp in history:              # 出现过 → 振荡
            return { stable: false, reason: "oscillation" }
        history.push(fp); lastFingerprint = fp
        applyState(circuit, next)
    return { stable: false, reason: "max_iter" }   # 超过 20 轮仍未稳
```

- **稳定**：连续两轮指纹相同 → 输出最终状态。
- **振荡**：指纹在历史里重复出现（在两个/多个状态间横跳）→ 典型互锁接错，提示：
  “电路状态无法稳定，可能存在互锁/反馈接线错误，请检查触点接线。”
- **超限**：达到 20 轮仍变化 → 同上提示。
- **确定性**：每轮先整体读旧状态、整体算、整体写新状态（不要边算边改），保证与遍历顺序无关。

> 自锁收敛示例：① 按下启动→线圈得电；② 线圈得电→辅助常开闭合；③ 松开启动→经辅助常开维持得电，指纹不变→稳定。

## 7. 故障态与复位

| 故障 | 触发 | 后果 | 复位方式 |
|---|---|---|---|
| 短路 | §5.3 命中 | 阻止该回路判为运行；红色危险提示 | 用户改线消除短路通路后重新运行 |
| 熔断器熔断 | 第一版手动触发（教学展示，不自动算电流） | 该边断开 | 右键“更换熔体” |
| 热继过载动作 | 第一版手动触发 / 模板脚本触发 | 主回路 + 控制常闭一起断开 | 排除后点击“复位” |

- 进入故障态的元件，其边在 §2.2 一律按“断开”处理。
- 复位前该回路保持断开，避免“边报短路边运行”。

## 8. 判错检查（拓扑级）

引擎对外提供 `check(circuit, expected?)`。分两层（对应 PRD §12）：

**通用检查（任何模式）——只看拓扑，不看意图：**

1. 悬空端子：存在 pin 不属于任何含 ≥2 pin 的节点（即没接线）。
2. 断路：练习要求工作的负载，其两端无法分别到达电源两极（§5.2 不成立）。
3. 短路：§5.3 命中。
4. 电源未接入：图中无任何电源极，或负载与电源极完全不连通。

**语义判错（仅练习模式，需 `expected` 模板）——对照预期结构：**

5. 元件类型错误：模板要求常闭处用户放了常开（如停止按钮 / 自锁触点）。
6. 拓扑错误：自锁触点应与启动按钮**并联**（两者各自两端分别同节点），用户接成**串联**（首尾相接成链）→ 报错。
7. 缺元件：模板要求的关键元件（如自锁辅助常开触点）在用户电路中缺失。

> 并联/串联的判定基于节点：A、B 两个双端元件**并联** ⟺ A 的两端节点集合 == B 的两端节点集合；**串联** ⟺ 它们恰好共享一个节点且各自另一端不同。

## 9. 输出（供 UI 渲染）

```ts
interface SimResult {
  stable: boolean;
  reason?: "oscillation" | "max_iter";
  energizedNodes: NodeId[];        // 带电节点 → 导线绿色高亮
  components: {                    // 每个元件的最终可视状态
    id: string;
    working?: boolean;            // 灯亮 / 电机转 / 线圈吸合
    closed?: boolean;            // 触点闭合
    faulted?: boolean;
  }[];
  shorts: { nodes: [NodeId, NodeId] }[];   // 短路对 → 红色闪烁
  errors: CheckError[];           // §8 判错结果
}
```

UI 颜色映射（与 PRD §15.3 运行模式一致）：带电导通=绿；未带电/断开=灰；错误=橙闪；短路危险=红闪。

## 10. 复杂度与边界

- 单轮：O(P + W + E)，E=元件边数；收敛最多 20 轮 → 整体仍近线性，第一版规模（几十个元件）毫秒级。
- 自环：wire 两端是同一 pin → 在生成节点前丢弃并提示“不能连接同一端子”。
- 重复连线：相同 pin 对之间多条 wire → 去重，不影响节点合并。
- 多电源极冲突（L 与另一电源 N 直连）→ 走 §5.3 短路。
- 无负载的纯回路（电源经开关直接回零线）→ §5.3 短路。

## 11. 验收用最小测试集

| 用例 | 期望 |
|---|---|
| L→开关(合)→灯→N | 灯亮 |
| L→开关(断)→灯→N | 灯灭 |
| L→灯（不接 N） | 灯灭 + 断路提示 |
| L→导线→N（无负载） | 短路提示，不判运行 |
| 点动：按住启动→线圈得电→主触点通→电机转；松开→停 | 通过 |
| 自锁：按启动→保持运行；按停止→停 | 通过，且收敛稳定 |
| 自锁缺辅助常开触点 | 松开即停 + 语义提示（练习模式） |
| 自锁触点接成串联 | 无法保持 + 语义提示（练习模式） |
| 互锁接错形成反馈 | 收敛返回 oscillation + 提示 |
| 热继动作 | 控制常闭断、线圈失电、电机停 |
